1. C/C++

~137 din 510 întrebări

Pointeri & tablouri (cel mai frecvent subiect, ~72 întrebări)

  • Un tablou int v[5] "decade" la un pointer către primul element când e folosit în expresii: v ≡ &v[0].
  • *(p+i) ≡ p[i] — aritmetica pointerilor avansează cu sizeof(tip) octeţi, nu cu 1 octet.
  • Pointer la pointer (int **pp) — folosit des la tablouri de şiruri (char *v[], indexare dublă v[i][j]).
  • Pointer către o structură auto-referenţială e OBLIGATORIU pointer (struct Nod *urm;), nu instanţă directă (struct Nod urm; nu compilează — tip incomplet/dimensiune infinită).
int v[] = {5, 10, 15, 20, 25};
int *p = v + 1;          // p indica spre v[1] = 10
printf("%d", *(p + 2) - *p);  // *(p+2)=v[3]=20 ; *p=v[1]=10 -> afiseaza 10
Capcană: confuzia dintre "avansează cu N poziţii" (aritmetică de pointer, corectă) şi "avansează cu N octeţi" (greşit) — p+2 pe un int* sare 2 întregi, nu 2 octeţi.

Recursivitate (~30 întrebări)

  • Trebuie urmărită "desfăşurarea" apelurilor pas cu pas (stivă), apoi "revenirea" cu valorile calculate.
  • Recursivitate cu acumulator (parametru care duce rezultatul parţial) vs. recursivitate care combină rezultatul DUPĂ apelul recursiv (return n + f(n-1);).
  • Fără caz de bază corect → stivă infinită → eroare de execuţie ("stack overflow"), una din variantele de răspuns clasice.
int f(int n, int p) {          // acumulator p
    if (n == 1) return p;
    return f(n - 1, p * n);    // f(5,1) = 5! = 120
}

Struct / union (~26 întrebări)

  • struct: fiecare membru are propria zonă de memorie (dimensiunea = suma membrilor, +eventual padding).
  • union: toţi membrii suprapun aceeaşi zonă de memorie (dimensiunea = cea a celui mai mare membru). Scrierea unui membru "stropeşte" pe ceilalţi.
  • Parametru struct/union transmis prin valoare la o funcţie → funcţia primeşte o COPIE; modificările nu se văd în funcţia apelantă.
union test { int x; char arr[4]; };
union test t; t.x = 0;               // toti cei 4 octeti deveni 0
t.arr[1]='G'; t.arr[2]='O'; t.arr[3]='L';
printf("%s", t.arr);   // arr[0] e inca 0 (terminator) -> afiseaza sir gol
Capcană: un union cu un octet nescris rămas 0 (terminator de şir) face ca %s să afişeze un şir gol, nu conţinutul "aşteptat".

Variabile static într-o funcţie (~30 întrebări, adesea combinat cu alte teme)

  • O variabilă static locală se iniţializează o singură dată şi îşi PĂSTREAZĂ valoarea între apeluri succesive ale funcţiei (spre deosebire de o variabilă locală obişnuită, care se reiniţializează la fiecare apel).

Şiruri de caractere funcţii din <string.h>

  • strlen nu numără terminatorul '\0'; sizeof pe un tablou de char ÎL include.
  • strcmp(a,b): 0 dacă egale, negativ dacă a<b lexicografic, pozitiv dacă a>b (compară codurile ASCII caracter cu caracter).
  • strcat scrie ÎNCEPÂND de la terminatorul nul al primului şir (deci indexul unde începe concatenarea = strlen al şirului original, înainte de apel).

Operatori pe biţi

OperatorEfect
<< / >>shift stânga/dreapta = înmulţire/împărţire cu 2 la puterea N (pt. întregi fără semn)
&, |, ^AND, OR, XOR pe biţi (nu confunda cu &&/|| logici)
Capcană: #define DUBLU(x) x + x fără paranteze — DUBLU(3) * 2 se expandează textual la 3 + 3 * 2 = 9, nu 12, din cauza priorităţii operatorilor.

Precedenţă & efecte secundare / comportament nedefinit

  • i++ (post-incrementare) în cadrul unei expresii returnează valoarea DINAINTE de incrementare; ++i o returnează pe cea de DUPĂ.
  • Ordinea de evaluare a argumentelor unei funcţii (ex. printf("%d %d", i++, i++)) NU este specificată de standard — grilele "curate" evită acest caz; dacă apare într-un subiect vechi cu ambiguitate reală, poate exista o eroare de barem.
  • Parametri impliciţi C++ (int f(int a, int b=5)) — valoarea implicită se foloseşte doar dacă parametrul lipseşte la apel.

Pointeri la funcţii

int inc(int x){ return x+1; }
int dbl(int x){ return x*2; }
int (*v[2])(int) = { inc, dbl };  // tablou de pointeri la functii
printf("%d", v[1](v[0](3)));     // dbl(inc(3)) = dbl(4) = 8

Declaraţia corectă a unui pointer la funcţie necesită paranteze în jurul numelui: int (*p)(int,int), nu int *p(int,int) (care declară o funcţie ce returnează int*).

2. Programare orientată pe obiecte (C++)

~118 din 510 întrebări — a doua categorie ca mărime

Clase & încapsulare

  • Specificatori de acces membri: private (implicit la class) / protected / public.
  • Metodă declarată const promite că nu modifică membrii obiectului; un obiect const poate apela DOAR metode const.
  • Pointerul this — pointer implicit către obiectul curent, util când parametrul unei metode are acelaşi nume ca un membru (this->x = x;).

Constructori şi destructori (~49 întrebări)

  • Ordinea la construcţie: mai întâi clasele de bază (în ordinea listată la moştenire), apoi membrii (în ordinea DECLARĂRII lor în clasă — nu ordinea din lista de iniţializare!), apoi corpul constructorului clasei derivate.
  • Ordinea la distrugere: exact inversă — întâi derivatul, apoi membrii (ordine inversă declarării), apoi bazele (ordine inversă moştenirii).
  • Constructor de copiere implicit face copiere superficială ("shallow" — copiază pointerii ca atare, nu ce indică ei); pentru copiere în profunzime ("deep") trebuie definit explicit.
  • Dacă o clasă are un constructor cu parametru definit explicit, compilatorul NU mai generează automat un constructor implicit fără parametri.
class Baza { public: Baza(){cout<<"B";} ~Baza(){cout<<"b";} };
class Derivat : public Baza { public: Derivat(){cout<<"D";} ~Derivat(){cout<<"d";} };
int main(){ Derivat x; }   // afiseaza: B D  (constructie) apoi d b (distrugere) -> "BDdb"

Moştenire (~25 întrebări)

  • class Derivat : Baza (fără specificator) → moştenire privată implicit (regulă specifică lui class; la struct implicitul e public).
  • Moştenire public: membrii public/protected ai bazei rămân cu acelaşi acces în derivat. Moştenire private: toţi membrii moşteniţi devin private în derivat.
  • Moştenire multiplă (class C : public A, public B) — ambiguitatea unui nume prezent în ambele baze se rezolvă explicit cu A::membru / B::membru.
  • Object slicing: atribuirea unui obiect derivat către o variabilă (nu pointer/referinţă) de tip bază "taie" partea derivată — rămâne doar partea de bază, iar apelurile ulterioare NU mai sunt polimorfice.

Polimorfism & funcţii virtuale (~51 întrebări — cel mai testat subiect din POO)

  • Apelul unei metode virtual printr-un pointer/referinţă de tip bază execută versiunea DIN CLASA REALĂ a obiectului (legare dinamică), indiferent de tipul static al pointerului.
  • O metodă NEvirtuală cu acelaşi nume într-o derivată doar "ascunde" (hiding) metoda bazei — apelul prin pointer de bază execută tot versiunea bazei (legare statică).
  • Destructor virtual: dacă un obiect derivat e şters (delete) printr-un pointer de tip bază şi destructorul bazei NU e virtual, se apelă DOAR destructorul bazei (scurgere/comportament incomplet). Cu destructor virtual, se apelă corect cel al derivatului, apoi al bazei.
  • Apelarea unei metode virtuale DIN constructorul/destructorul bazei NU e polimorfică acolo — foloseşte versiunea clasei curente în construcţie (obiectul derivat încă nu e complet construit / e deja parţial distrus).

Clase abstracte (~14 întrebări)

  • O clasă cu cel puţin o metodă virtual ... = 0; (virtuală pură) devine abstractă.
  • Nu se pot instanţia obiecte de tip abstract, dar SE POT declara pointeri/referinţe de acel tip.
  • O clasă derivată dintr-o clasă abstractă rămâne ea însăşi abstractă dacă nu suprascrie TOATE metodele virtuale pure moştenite.
  • O clasă abstractă POATE avea date membre, constructori şi metode nevirtuale — restricţia e doar asupra instanţierii directe.

Supraîncărcarea operatorilor

  • Ca metodă membru: primul operand e obiectul însuşi (this); util pt. operator+, operator==, operator[].
  • Ca funcţie friend: necesară când primul operand NU e de tipul clasei (ex. 50 + obiect) sau pentru operator<< de afişare (primul operand e ostream&).
  • Operatori care NU pot fi supraîncărcaţi: . , .* , :: , ?: , sizeof.

Funcţii şi clase friend

O funcţie/clasă friend are acces la membrii private/protected ai clasei care o declară astfel, deşi nu e membră a clasei respective. friend NU se moşteneşte şi nu e reciprocă.

Membri statici & template

  • Un membru static al unei clase e PARTAJAT de toate instanţele (o singură copie, indiferent câte obiecte se creează).
  • template<typename T> permite scrierea unei funcţii/clase generice, instanţiată de compilator pentru tipul concret folosit la apel.

3. SQL (Oracle)

~114 din 510 întrebări

LDD — Limbajul de Definire a Datelor (DDL)

  • Comenzi: CREATE, ALTER, DROP, TRUNCATE.
  • Fiecare comandă LDD face automat COMMIT — atât asupra propriilor modificări, cât şi asupra oricăror modificări LMD nesalvate anterior în aceeaşi sesiune.
Capcană clasică: un INSERT urmat de o comandă LDD (ex. CREATE VIEW, ALTER TABLE) apoi de ROLLBACK — ROLLBACK-ul NU mai are ce anula, pentru că LDD-ul a făcut deja commit implicit înaintea sa.

LMD — Limbajul de Manipulare a Datelor (DML)

SELECT, INSERT, UPDATE, DELETE — modificările NU sunt permanente până la COMMIT (pot fi anulate cu ROLLBACK).

Constrângeri & chei străine

ConstrângereRol
PRIMARY KEYunicitate + NOT NULL, o singură cheie primară / tabelă
FOREIGN KEY ... REFERENCESvaloarea trebuie să existe în tabela părinte (sau să fie NULL)

  • ON DELETE CASCADE: ştergerea rândului părinte şterge şi rândurile copil.
  • ON DELETE SET NULL: ştergerea părintelui pune NULL pe cheia străină a copiilor (nu îi şterge).
  • Fără nicio clauză ON DELETE: ştergerea părintelui cu copii existenţi generează EROARE (încălcare integritate referenţială).

JOIN-uri (~20 întrebări)

  • INNER JOIN: doar perechile care se potrivesc pe condiţie.
  • LEFT JOIN: toate rândurile din tabela stângă, chiar fără corespondent (coloanele din dreapta devin NULL).
  • Self-join: o tabelă joncţionată cu ea însăşi (alias diferit), clasic pentru relaţii angajat–şef (id_sef care referă id_angajat din aceeaşi tabelă).
  • COUNT(coloană) pe un LEFT JOIN numără doar valorile NEnule ale acelei coloane → util pt. "0 subordonaţi" în loc de a exclude rândul.

Subinterogări (~24 întrebări)

  • = ANY (subcerere) ≡ IN (subcerere).
  • <> ALL (subcerere) ≡ NOT IN (subcerere).
  • EXISTS întoarce adevărat/fals (nu un număr) — dacă subcererea produce cel puţin un rând.
Capcana NOT IN + NULL: dacă subcererea din NOT IN (...) întoarce fie şi o singură valoare NULL, întreaga condiţie devine NULL pentru TOATE rândurile → interogarea nu întoarce niciun rând (nu eroare, doar rezultat gol). NOT EXISTS nu are această problemă.

Funcţii de grup, GROUP BY / HAVING (~38 întrebări)

  • WHERE filtrează rândurile ÎNAINTE de grupare; HAVING filtrează grupurile DUPĂ agregare — HAVING poate folosi funcţii de grup (COUNT, SUM...), WHERE nu.
  • Funcţiile de grup (COUNT, AVG, SUM...) IGNORĂ valorile NULL, cu excepţia COUNT(*) care numără toate rândurile.
  • O comandă SELECT cu funcţii de grup dar FĂRĂ GROUP BY tratează întreaga tabelă filtrată ca un singur grup (întoarce un singur rând agregat).

Operatori mulţime

Vederi (views)

  • O vedere simplă (un singur SELECT fără GROUP BY/DISTINCT/joncţiune complexă) e actualizabilă direct prin INSERT/UPDATE/DELETE — modificarea se propagă în tabela de bază.
  • WITH CHECK OPTION: interzice ca un UPDATE/INSERT prin vedere să producă un rând care nu mai satisface condiţia WHERE a vederii (eroare, nu execuţie silenţioasă).
  • O vedere pe bază de joncţiune/agregare, neactualizabilă direct, poate fi făcută "actualizabilă" printr-un trigger INSTEAD OF.

Funcţii utile Oracle

LCT — Limbajul de Control al Tranzacţiilor

  • COMMIT: face permanente modificările LMD ale sesiunii curente, vizibile altor sesiuni.
  • ROLLBACK: anulează toate modificările LMD nesalvate din tranzacţia curentă.
  • SAVEPOINT nume + ROLLBACK TO nume: anulează doar modificările făcute DUPĂ acel savepoint, păstrându-le pe cele dinainte.

4. PL/SQL

~117 din 510 întrebări

Structura unui bloc

DECLARE
    variabila NUMBER;      -- optional
BEGIN
    -- instructiuni executabile
EXCEPTION
    WHEN altceva THEN
        -- tratare erori
END;
/

DECLARE e opţional (un bloc poate fi doar BEGIN...END; dacă nu declară nimic). / pe o linie separată e comanda SQL*Plus care trimite blocul spre execuţie.

Cursor implicit

  • Orice comandă SELECT INTO / UPDATE / DELETE / INSERT foloseşte automat un cursor implicit, gestionat de Oracle (deschis şi închis automat).
  • Atribute: SQL%ROWCOUNT (câte rânduri afectate), SQL%FOUND/SQL%NOTFOUND.
  • SELECT ... INTO care nu găseşte niciun rând ridică NO_DATA_FOUND; care găseşte mai multe rânduri ridică TOO_MANY_ROWS.

Cursor explicit (~28 întrebări menţionează "cursor")

CURSOR c IS SELECT nume FROM angajati WHERE id_departament=50;
v angajati.nume%TYPE;
BEGIN
    OPEN c;
    FETCH c INTO v;          -- primul fetch, INAINTE de verificarea %FOUND
    WHILE c%FOUND LOOP       -- idiom CORECT: fetch, apoi verifica
        DBMS_OUTPUT.PUT_LINE(v);
        FETCH c INTO v;
    END LOOP;
    CLOSE c;
AtributValoare înainte de primul FETCHDupă
%FOUNDNULL (necunoscut)TRUE/FALSE dacă ultimul fetch a adus un rând
%NOTFOUNDNULLopusul lui %FOUND
%ISOPENFALSE (înainte de OPEN)TRUE cât timp e deschis
%ROWCOUNT0numărul cumulat de fetch-uri reuşite
Capcana cea mai frecventă din tot setul de grile: o buclă WHILE cursor%NOTFOUND LOOP ... FETCH ... scrisă ÎNAINTE de orice FETCH nu execută NICIODATĂ corpul buclei, pentru că %NOTFOUND e NULL (tratat ca FALSE într-un WHILE), nu FALSE propriu-zis şi nici TRUE. Idiomul corect e "fetch înaintea buclei" (ca mai sus) sau bucla FOR ... IN (SELECT...) care gestionează asta automat.

Cursor FOR loop — cea mai simplă formă, deschide/preia/închide cursorul automat:

FOR rang IN (SELECT * FROM angajati WHERE salariul > 5000) LOOP
    DBMS_OUTPUT.PUT_LINE(rang.nume);
END LOOP;   -- nu necesita DECLARE separat pt cursor, nici OPEN/FETCH/CLOSE

%TYPE / %ROWTYPE

  • variabila tabelă.coloană%TYPE — variabilă cu acelaşi tip ca acea coloană (nu invers: coloană.tabelă e greşit).
  • variabila tabelă%ROWTYPE — o înregistrare (record) cu câte un câmp pentru FIECARE coloană a tabelei (acces cu v.nume_coloana).

Excepţii (~22 întrebări)

Excepţie predefinităCauză
NO_DATA_FOUNDSELECT INTO fără niciun rând găsit
TOO_MANY_ROWSSELECT INTO cu mai multe rânduri decât o variabilă simplă
ZERO_DIVIDEîmpărţire la zero
DUP_VAL_ON_INDEXîncălcare unicitate (cheie primară/UNIQUE)
  • Excepţie definită de utilizator: se declară (e_mea EXCEPTION;), se ridică (RAISE e_mea;), se tratează (WHEN e_mea THEN).
  • RAISE_APPLICATION_ERROR(-20000..-20999, 'mesaj') — modul standard de a semnala o eroare aplicativă cu cod şi mesaj propriu.
  • O excepţie tratată LOCAL (bloc/procedură cu propriul EXCEPTION WHEN) NU se propagă mai departe — blocul apelant continuă normal după apel.
  • WHEN OTHERS trebuie să fie ULTIMA ramură dintr-un EXCEPTION (altfel eroare de compilare).

Proceduri & funcţii stocate (~17 întrebări despre parametri IN/OUT)

  • Parametru IN (implicit): doar citire — atribuirea unei valori în corpul procedurii e eroare de compilare (PLS-00363).
  • OUT: doar scriere, valoarea iniţială la apel se pierde (parametrul e NULL la intrare).
  • IN OUT: se poate citi ŞI scrie.
  • O funcţie apelabilă dintr-o comandă SQL (SELECT) NU poate executa DML (INSERT/UPDATE/DELETE) — genera eroare ORA-14551, cu excepţia cazului în care acel DML e cod mort (ex. plasat după un RETURN).
  • Apelarea cu notaţie numită: proc(param2 => val2, param1 => val1) — ordinea nu mai contează.

Triggere (~14 întrebări)

  • BEFORE / AFTER + INSERT/UPDATE/DELETE, la nivel de instrucţiune (implicit) sau FOR EACH ROW (nivel de rând).
  • :NEW / :OLD disponibile DOAR la triggere FOR EACH ROW; :OLD nu există la INSERT, :NEW nu există la DELETE.
  • :NEW poate fi MODIFICAT doar într-un trigger BEFORE (nu AFTER — rândul e deja scris în tabelă).
  • Trigger INSTEAD OF: definit pe o VEDERE, înlocuieşte operaţia DML implicită cu cod propriu (util pt. vederi neactualizabile direct).

Logica NULL în PL/SQL (NULL apare în ~78 de întrebări per total)

Regula de bază, testată constant: orice comparaţie cu NULL (v > 0, v = 5 etc.) dă NULL (necunoscut), NICIODATĂ TRUE. Într-un IF/WHILE, condiţia NULL e tratată ca FALSE — se execută ramura ELSE (dacă există), FĂRĂ nicio eroare de execuţie.
v_nr NUMBER;              -- neinitializata = NULL
BEGIN
  IF v_nr > 0 THEN DBMS_OUTPUT.PUT_LINE('POZITIV');
  ELSE DBMS_OUTPUT.PUT_LINE('ALTCEVA'); END IF;
  -- afiseaza 'ALTCEVA', nu eroare, pentru ca (NULL > 0) e NULL -> ramura ELSE

5. Baze de date — teorie generală

~24 din 510 întrebări (categorie separată), plus concepte reluate în SQL

Modelul entitate-relaţie

  • Entitate (obiect din lumea reală), atribut (proprietate), relaţie/asociere (legătură între entităţi).
  • Cardinalitate: 1:1, 1:N, M:N. O relaţie M:N se implementează în modelul relaţional printr-o TABELĂ DE LEGĂTURĂ (asociativă), cu chei străine către ambele entităţi.
  • Asociere recursivă: relaţie a unei entităţi cu ea însăşi (ex. angajat–şef).

Chei (~20 întrebări)

TermenDefiniţie
Supercheieorice mulţime de atribute care identifică unic un rând (poate conţine atribute redundante)
Cheie candidată (minimă)supercheie MINIMĂ — fără niciun atribut care poate fi eliminat păstrând unicitatea
Cheie primarăcheia candidată ALEASĂ pentru a identifica rândurile tabelei; unică per tabelă
Cheie alternativăorice cheie candidată NEALEASĂ ca primară
Cheie străinăatribut(e) care referă cheia primară a altei tabele (sau a aceleiaşi, la asociere recursivă)

Orice cheie candidată e o supercheie, dar nu orice supercheie e cheie candidată (doar cele minime).

Normalizare & anomalii (forme normale + anomalii)

  • 1NF: valori atomice (fără grupuri repetitive/atribute multivaloare).
  • 2NF: 1NF + fiecare atribut non-cheie depinde de ÎNTREAGA cheie primară (nu doar de o parte, la chei compuse).
  • 3NF: 2NF + fără dependenţe tranzitive (un atribut non-cheie nu depinde de alt atribut non-cheie).
  • O schemă nenormalizată (date repetate) suferă de: anomalie de inserare (nu poţi adăuga o informaţie fără să ai deja alta asociată), anomalie de actualizare (aceeaşi informaţie trebuie modificată în mai multe rânduri, risc de inconsistenţă) şi anomalie de ştergere (ştergerea unui rând pierde şi altă informaţie, încă necesară).

Proprietăţile ACID ale tranzacţiilor

  • Atomicitate — o tranzacţie se execută în întregime sau deloc.
  • Consistenţă — trece baza de date dintr-o stare validă în alta validă.
  • Izolare — tranzacţiile concurente nu se interferează vizibil una cu alta.
  • Durabilitate — odată confirmată (COMMIT), o modificare rezistă chiar şi la o cădere de sistem.
